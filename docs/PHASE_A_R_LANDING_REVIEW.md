### What likely caused the regressions (hypothesis)

`home-v2.css` is bundled into the main CSS chunk (Home.tsx imports it eagerly via the router). All its rules are properly prefixed `.home-shell-v2 ...` — verified by audit + the R-Landing spec assertion #6 (`/login` color is NOT home ink, scope did not leak in COMPUTED STYLE).

**However**, the +3705 line CSS file introduces a LOT of new rules into the cascade. Several specs assert specific computed styles or layout positions on workspace/auth routes:
- R5 test #3: brand panel `max-height: 260px` at viewport 768 — relies on `@media (max-width: 880px)` matching. Production CSS rule is present in the bundle; assertion fails with brand panel measuring 1085px.
- R1.1 test #9: hamburger `#appshell-sidebar-trigger` HIDDEN at ≥1024px workspace — relies on R7.12 CSS. Production CSS rule is present; assertion fails with hamburger visible.

**Hypothesis 1:** Vite CSS chunking reordered rules in a way that changes cascade ties.
**Hypothesis 2:** The 3705 new home-v2.css lines push the bundle past a CSS-parser limit or trigger an optimization that drops/reorders rules.
**Hypothesis 3:** Pre-existing flakes that R-Landing exposed by adding load to the bundle (slower CSS parse → longer first paint → test timing different).

The R-Landing spec (10/10 PASS) confirms the home redesign works correctly in isolation. The cross-route regression is real but needs deeper investigation — it's not a "leak from scoped CSS to other routes" per the computed-style assertion proof.

**Recommendation:** investigate as R-Landing-followup once owner gives D13 ack on the landing itself. The widespread regression pattern suggests a Vite/build-pipeline interaction; could be a 30-min investigation OR a deeper rabbit hole.
